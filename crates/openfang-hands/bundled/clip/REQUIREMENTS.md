# Clip Hand — 产品需求说明（PRD）
本文档定义智能剪辑相关能力范围、验收口径与分期目标。**以本文档为需求真源**。

- **Hand ID `clip`（Clip Hand）**：稳定基线，既有行为尽量保持不变。
- **Hand ID `autoclip`（AutoClip Hand）**：升级版实现载体，与 `clip` 并行；在 OpenFang 内为 Bundled Hand（`crates/openfang-hands/bundled/autoclip/`），V1 复用同一套 shell/ffmpeg 流水线并逐步按本文档演进（VLM、素材缺口等）。

后续也可拆出独立服务或 CLI，与上述 Hand 共用需求条目。
---
## 1. 愿景与范围
### 1.1 一句话目标
**自动完成「理解素材 → 决策剪辑 →（可选）补足素材 → 导出成片」**，在可控人工介入下尽量接近目标平台与目标受众的**最优传播效果**。
### 1.2 不在首版范围（需显式另立项目或后续版本）
- 替代专业 NLE（多轨精剪、复杂调色、电影级混音）的完整工具链。
- 无版权审查策略的「任意联网抓素材」（须单独定义合规与来源白名单）。
---
## 2. 核心需求（四条）
### R1 — 自动智能剪辑（核心）
**描述**  
系统应在**默认配置**下，将输入素材处理为约定形态的输出（如多条竖屏短片或单条主片），包括：时间线决策、切点、基础包装（比例、字幕、响度等按版本定义）。
**验收要点**
- 给定标准测试素材集，可无人工逐步指导完成端到端导出。
- 输出路径、命名、数量符合配置与报告约定。
- 失败时有可诊断错误（依赖缺失、转写失败、编码失败等），而非静默烂尾。
**现状（Clip Hand）**  
具备：URL/本地 → 转写（多路径）→ 选段 → FFmpeg 切条、9:16、烧字幕、缩略图、可选 TTS、可选发布。  
缺口：叙事型「单主片」多段拼接、多轨；「智能」深度受限于以文稿为主的分析。
---
### R2 — 自行分析给定素材
**描述**  
对用户**已提供**的素材（视频/音频及落盘后的文件；链接经下载后等同）进行**结构化理解**，支撑后续剪辑决策。
**最低能力（V1）**
- 元数据：时长、分辨率、是否有音频轨等（`ffprobe` 级）。
- 语音内容：带时间轴的转写（词级/句级优先），统一为内部 `transcript` 结构。
- 文本推理：基于转写生成候选高光区间、标题与理由（结构化 JSON）。
**增强能力（V2+）**
- 画面/视频理解：关键帧或短片断 + VLM，输出与时间点对齐的视觉摘要（演示、人物、场景类型等）。
- 质量信号：静音段、场景切换点、黑场等，用于微调切点与节奏。
**验收要点**
- 分析结果可序列化为机器可读格式（如 JSON），供选段与导出模块消费。
- 同一素材重复运行，在确定性配置下结果可复现或可解释差异来源（模型温度、外部 API 等）。
**现状**  
V1 中文稿路径较完整；V2+ 视觉理解未纳入工具与 prompt 契约。
---
### R3 — 自行寻找还需要的素材
**描述**  
在分析阶段识别**内容缺口**（如 B-roll、补镜、配乐、音效、封面图等），并按策略**获取或生成**可用素材，再与主时间线合并。
**能力分解**
- **规划**：输出 `gaps[]`（类型、建议时长、风格、关键词、插入位置约束）。
- **获取**：可插拔适配器——内部素材库、授权外部 API、生成模型、用户二次上传等。
- **合规**：每条素材须记录来源与许可类型；默认禁止未授权抓取。
**验收要点**
- 无可用来源时，系统降级为「仅主素材剪辑」并明确提示缺口未满足。
- 合并后成片可播放，音画不同步、分辨率突变等有明确处理策略（scale/pad/限幅）。
**现状**  
Clip Hand **未实现**该闭环；`web_fetch` 等通用能力未定义为「缺素材 → 检索 → 下载 → 许可检查」流程。
---
### R4 —「最佳」视频剪辑
**描述**  
「最佳」必须**可操作化**：面向平台、时长、钩子强度、节奏、品牌风格等形成**目标函数或规则集**，使系统能比较候选方案并输出推荐结果。
**V1 建议指标（示例，可配置）**
- 首 3 秒钩子（文稿或画面摘要是否满足关键词/强度阈值）。
- 单片时长落在配置区间（如短视频 30–90s）。
- 切点优先落在停顿、场景切换或静音谷（与 R2 信号结合）。
- 响度归一或峰值限制（避免导出忽大忽小）。
**V2+**
- 多候选时间线自动生成 + 粗评分 + 推荐一条。
- 可选人机闭环：用户对版本打分反哺权重。
**验收要点**
- PRD 中写明默认平台与默认指标；验收用例与评分记录可追溯。
- 「最佳」不作为无法定义的形容词，而对应上述可测项。
**现状**  
有部分传播向启发式（选段规则）；**缺**统一评分与多候选对比机制。
---
## 3. 分期路线图（与四条需求对应）
| 阶段 | 目标 | 覆盖需求 |
|------|------|-----------|
| **V1** | 单输入 → 转写 → 结构化选段 → FFmpeg 导出（竖屏+字幕为主）；可选单一 STT/TTS 路径收敛 | R1、R2（基线） |
| **V2** | 引入 VLM/关键帧管线，融合文稿+画面选段；切点规则与简单「最佳」评分 | R2（增强）、R4（初版） |
| **V3** | Gap 规划 + 素材适配器（内网库/授权 API/用户上传）；合规元数据进时间线 | R3 |
| **V4** | 多候选时间线、主片叙事拼接、更强平台模板 | R1、R4（进阶） |
---
## 4. 接口与工件约定（实现无关）
以下名称供 HAND / 独立工具 / 服务共用，便于替换实现。
| 工件 | 说明 |
|------|------|
| `source.*` | 归一化后的主视频（工作目录内） |
| `transcript.json` | 带时间轴的转写（内部标准结构） |
| `analysis.json` | 可选：章节、视觉摘要、质量信号 |
| `gaps.json` | 可选：素材缺口描述（R3） |
| `segments.json` | 选段结果：`start_sec` / `end_sec` / `title` / `rationale` 等 |
| `timeline.json` | 可选：多段拼接、插入 B-roll 后的主时间线 |
| 导出目录 | `clip_*_final.mp4`、SRT、缩略图等 |
---
## 5. 非功能需求（摘要）
- **跨平台**：Windows / macOS / Linux 下路径与 ffmpeg 滤镜路径规则一致且可测。
- **安全**：API Token 不得写入用户可见报告；临时文件清理策略明确。
- **可观测**：关键阶段耗时、失败步骤、使用的 STT/TTS/发布渠道可记录（不含密钥）。
